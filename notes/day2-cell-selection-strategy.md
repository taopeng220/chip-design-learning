# 标准单元选择策略 - FusionCompiler vs Yosys

📅 日期：2026-03-12
🎯 理解工具如何在多个候选单元中做选择

---

## 🎭 场景：选择 AND2 门

假设工艺库有这些选择：

| 单元 | 延迟 (ns) | 面积 (um²) | 驱动能力 | 功耗 (uW) |
|------|-----------|-----------|---------|-----------|
| AND2_X1 | 0.50 | 2.0 | 低 (1x) | 10 |
| AND2_X2 | 0.30 | 3.0 | 中 (2x) | 18 |
| AND2_X4 | 0.20 | 4.5 | 高 (4x) | 30 |

**问题：** 实现 `y = a & b`，选哪个？

---

## ⚖️ Trade-off 分析

### 没有"完美"的选择！

```
  快速 ←─────────────→ 省面积
    ↑                    ↑
AND2_X4              AND2_X1
(贵、耗电)          (慢、驱动弱)
         ↑
     AND2_X2
    (折中方案)
```

### 关键因素：

#### 1️⃣ **时序要求**
```
IF 这个门在关键路径上（Critical Path）:
    必须用 AND2_X4（快）
ELSE IF 有一定时序裕量（Slack > 0.2ns）:
    可以用 AND2_X1（省）
ELSE:
    用 AND2_X2（平衡）
```

#### 2️⃣ **扇出负载**
```
IF 这个门驱动 10+ 个其他门:
    必须用 AND2_X4（驱动强）
ELSE IF 只驱动 1-2 个门:
    用 AND2_X1 就够（不浪费）
ELSE:
    用 AND2_X2
```

#### 3️⃣ **优化目标**
```
IF 优化目标 = 最快速度（high performance）:
    全部用 X4
ELSE IF 优化目标 = 最小面积（area-critical）:
    尽量用 X1
ELSE IF 优化目标 = 低功耗（low power）:
    尽量用 X1
ELSE:
    平衡优化
```

## 🔧 FusionCompiler 的决策流程

### Step 1: 初始映射（保守）
```tcl
compile
```

**策略：**
- 先用**中等驱动**单元（X2）
- 确保功能正确
- 建立初始时序

### Step 2: 时序分析
```tcl
report_timing
```

**发现问题：**
- 某些路径违反时序（Slack < 0）
- 某些路径有很大裕量（Slack > 1ns）

### Step 3: 增量优化
```tcl
compile_ultra -incremental
```

**优化动作：**

#### 对于关键路径（Critical Path）：
```
原来: AND2_X2 (delay=0.3ns, slack=-0.1ns) ❌
优化: AND2_X4 (delay=0.2ns, slack=+0.0ns) ✅

动作: UPSIZE (增加驱动能力)
```

#### 对于裕量充足的路径：
```
原来: AND2_X2 (delay=0.3ns, slack=+0.8ns)
优化: AND2_X1 (delay=0.5ns, slack=+0.6ns) ✅

动作: DOWNSIZE (减小面积/功耗)
依然满足时序！
```

### Step 4: 全局平衡
```
FusionCompiler 会迭代优化：
1. 修复最差的时序违规
2. 回收不需要的资源（downsize）
3. 平衡面积/速度/功耗
4. 重复，直到收敛
```

## 🎓 Yosys/ABC 的决策流程

### 局限性：没有全局时序信息

```tcl
# Yosys 综合
synth -top design

# ABC 映射
abc -liberty tech.lib
```

**ABC 怎么选？**

### 默认策略：最小化延迟

```
对每个逻辑门：
    选择 delay 最小的单元（通常是 X4）

结果：
    ✅ 速度快
    ❌ 面积大（浪费）
    ❌ 功耗高（浪费）
```

### 优化面积策略：

```tcl
abc -liberty tech.lib -A
```

```
对每个逻辑门：
    选择 area 最小的单元（通常是 X1）

结果：
    ✅ 面积小
    ❌ 可能时序不满足
    ❌ 驱动能力可能不足
```

### 为什么 Yosys 不够智能？

**因为它缺少关键信息：**
- ❌ 没有全局时序分析（STA）
- ❌ 不知道哪些路径是关键路径
- ❌ 不知道扇出负载
- ❌ 没有迭代优化

**Yosys 的定位：**
- 做**逻辑综合**和**初始映射**
- 后续交给 **OpenROAD** 做物理感知优化
- 分工合作！

## 📊 实战例子对比

### 设计：4-bit 加法器

```verilog
module adder_4bit(
    input [3:0] a, b,
    input cin,
    output [3:0] sum,
    output cout
);
    assign {cout, sum} = a + b + cin;
endmodule
```

### FusionCompiler 综合结果

```verilog
// 假设时钟频率：500MHz（周期 2ns）

module adder_4bit(...);
    wire c0, c1, c2;

    // Bit 0（最早，裕量大）
    XOR2_X1 u1 (.A(a[0]), .B(b[0]), .Z(s0));     // 用 X1（省）
    XOR2_X1 u2 (.A(s0), .B(cin), .Z(sum[0]));

    // Bit 1（中间）
    XOR2_X2 u3 (.A(a[1]), .B(b[1]), .Z(s1));     // 用 X2（平衡）

    // Bit 3（关键路径，最晚）
    XOR2_X4 u10 (.A(a[3]), .B(b[3]), .Z(s3));    // 用 X4（快）
    XOR2_X4 u11 (.A(s3), .B(c2), .Z(sum[3]));    // 用 X4（快）

    // Carry 链（关键路径）
    AND2_X4 u4 (...);  // 全部用 X4，因为延迟累积
    OR2_X4  u5 (...);
    ...
endmodule
```

**特点：**
- ✅ 不同位置用不同驱动强度
- ✅ 关键路径用强驱动
- ✅ 非关键路径省面积
- ✅ 整体平衡优化

**统计：**
- 总面积：100 um²
- 关键路径延迟：1.8 ns（满足 2ns 时钟）
- 功耗：50 uW

---

### Yosys + ABC 综合结果（默认）

```verilog
module adder_4bit(...);
    // ABC 选择：全部最小延迟（X4）

    XOR2_X4 u1 (.A(a[0]), .B(b[0]), .Z(s0));     // X4（浪费）
    XOR2_X4 u2 (.A(s0), .B(cin), .Z(sum[0]));    // X4（浪费）
    XOR2_X4 u3 (.A(a[1]), .B(b[1]), .Z(s1));     // X4
    XOR2_X4 u10 (.A(a[3]), .B(b[3]), .Z(s3));    // X4
    AND2_X4 u4 (...);
    OR2_X4  u5 (...);
    ...
endmodule
```

**特点：**
- ✅ 速度最快
- ❌ 面积浪费（很多地方不需要 X4）
- ❌ 功耗浪费

**统计：**
- 总面积：150 um²（比 FC 多 50%）
- 关键路径延迟：1.5 ns（比需要的快）
- 功耗：80 uW（比 FC 多 60%）

---

### Yosys + ABC 综合结果（优化面积）

```tcl
abc -liberty tech.lib -A
```

```verilog
module adder_4bit(...);
    // ABC 选择：全部最小面积（X1）

    XOR2_X1 u1 (.A(a[0]), .B(b[0]), .Z(s0));     // X1
    XOR2_X1 u2 (.A(s0), .B(cin), .Z(sum[0]));    // X1
    XOR2_X1 u10 (.A(a[3]), .B(b[3]), .Z(s3));    // X1（不够）
    AND2_X1 u4 (...);  // X1（延迟大）
    OR2_X1  u5 (...);
    ...
endmodule
```

**特点：**
- ✅ 面积最小
- ✅ 功耗最低
- ❌ 关键路径延迟可能不满足时序

**统计：**
- 总面积：70 um²
- 关键路径延迟：2.5 ns（违反 2ns 时钟！）❌
- 功耗：35 uW

## 🎯 关键启示

### 1. FusionCompiler 的优势

```
全局优化 = 智能选择

✅ 时序驱动（Timing-driven）
✅ 知道哪里需要快，哪里可以省
✅ 迭代优化，达到最佳平衡
✅ 考虑扇出、负载、时序约束

结果：PPA 最优（Performance, Power, Area）
```

### 2. Yosys/ABC 的局限

```
局部优化 = 简单策略

❌ 只看单个门，不看全局
❌ 要么全快（浪费），要么全省（可能不够）
❌ 不知道时序约束

结果：需要后续工具（OpenROAD）进一步优化
```

### 3. 为什么还要用 Yosys？

**因为它是流程的一部分：**

```
Yosys: 逻辑综合 + 初始映射
   ↓
OpenROAD: 物理感知优化
   ↓  (这里会重新选择单元！)
   ↓  - 基于真实布局
   ↓  - 基于真实线长
   ↓  - 基于全局时序
   ↓
最终优化的网表
```

**Yosys 负责"初稿"，OpenROAD 负责"精修"！**

## 🤔 思考题

### Q1: 如果没有时序约束（SDC），FusionCompiler 会怎么选择单元？

**答案：**
- 默认优化面积（选 X1）
- 因为没有时序压力
- 除非用户指定其他优化目标

### Q2: OpenROAD 能做 FusionCompiler 做不到的事吗？

**答案：**
- 功能上：不能，FC 更强大
- 但 OpenROAD 是开源的：
  - ✅ 免费
  - ✅ 可以看源码学习算法
  - ✅ 可以定制修改
  - ✅ 学术研究友好

### Q3: 为什么不能先全部用 X4，后面再换成 X1？

**答案：**
- 可以，这叫 **"先快后省"策略**
- 但效率低（需要多次迭代）
- FusionCompiler 的策略：
  - 初始就用中等单元（X2）
  - 同时 upsize 和 downsize
  - 更快收敛

## 📚 总结

### 工艺映射的本质

```
选择标准单元 = 多目标优化问题

目标：
  - 满足时序 (delay)
  - 最小面积 (area)
  - 最低功耗 (power)

约束：
  - SDC 时序约束
  - 扇出/负载
  - 物理位置（后期）

策略：
  - FusionCompiler: 全局智能优化 ✨
  - Yosys/ABC: 局部简单策略
  - OpenROAD: 物理感知优化
```
